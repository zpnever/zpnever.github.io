Pertanyaan kunci: apakah itinerary AI via OpenAI/Gemini/Groq API masuk akal secara teknis dan bisnis? Jawaban singkat: ya, dan ini justru keputusan paling strategis — tapi dengan catatan penting soal timing dan cara implementasinya.

OpenAI GPT-4o / GPT-4o mini Direkomendasikan (fase 1)
Gunakan untuk generate itinerary natural language. User ketik "3 hari di Yogyakarta, budget 800ribu, suka kuliner dan budaya" — GPT menghasilkan rancangan perjalanan yang enak dibaca, kemudian Trabas inject data tiket & harga real dari database sendiri.

Kelebihan

  • Output paling natural & koheren
  • Function calling untuk structured data
  • GPT-4o mini sangat murah
  • Ekosistem dokumentasi terlengkap

Kelemahan

  • Data wisata Indonesia terbatas
  • Bisa "hallucinate" nama tempat
  • Biaya naik seiring scale
  • Data user ke server US
Google Gemini API Kondisional (jika butuh Maps)
Keunggulan terbesar Gemini adalah integrasi native dengan Google Maps dan Places API. Jika Trabas butuh data jarak, estimasi waktu tempuh, dan jam buka tempat wisata secara real-time, kombinasi Gemini + Google Maps adalah yang paling mulus.

Kelebihan

  • Native Google Maps integration
  • Gratis tier sangat generous
  • Multimodal (bisa analisis foto wisata)

Kelemahan

  • Kualitas Bahasa Indonesia masih di bawah GPT
  • API berubah-ubah (masih beta)
  • Double dependency ke Google
Groq API (Llama / Mixtral) Kondisional (untuk kecepatan)
Groq bukan model AI — ini adalah inference engine yang sangat cepat (ratusan token/detik). Cocok untuk fitur yang butuh respons instan seperti autocomplete destinasi atau saran cepat. Bukan untuk output kompleks seperti full itinerary 3 hari.

Kelebihan

  • Tercepat di industri saat ini
  • Harga sangat murah
  • Cocok untuk quick suggestions

Kelemahan

  • Model open-source, kualitas terbatas
  • Tidak cocok untuk output panjang
  • Kurang baik di Bahasa Indonesia
Fine-tuned model sendiri (jangka panjang) Target fase 3–4
Di fase 3, Trabas sudah punya ribuan data perjalanan pengguna nyata, rating destinasi, dan pola itinerary yang berhasil. Ini adalah training data emas yang tidak dimiliki kompetitor manapun. Fine-tune model open-source (Llama 3, Qwen) dengan data ini untuk membuat AI yang benar-benar paham konteks wisata Indonesia.
✓ Inilah yang membuat AI itinerary Trabas menjadi moat jangka panjang — bukan karena pakai API OpenAI (siapapun bisa), tapi karena model ditraining dengan data wisata Indonesia yang hanya dimiliki Trabas.
Arsitektur AI yang Direkomendasikan
  • Fase 1–2: OpenAI GPT-4o mini sebagai "brain", database Trabas sebagai "ground truth"
  • Prompt engineering: inject data tiket + jam buka + rating ke context window sebelum generate
  • Validasi output: filter hasil AI dengan data real sebelum ditampilkan ke user
  • Fase 3+: Mulai fine-tune model sendiri, kurangi ketergantungan third-party API
  • Biaya estimasi awal: ~$0.05–0.15 per itinerary request (GPT-4o mini)
Tinggi
Vendor tidak mau onboarding
Pengelola wisata lokal (terutama desa wisata) tidak punya smartphone memadai, tidak mengerti digital, atau tidak percaya platform baru.
→ Mitigasi: Tim field sales dedicated, bantuan device, komisi 0% 6 bulan pertama, pendampingan langsung.
Tinggi
Data tiket tidak akurat / kedaluwarsa
Harga berubah mendadak, tempat wisata tutup tiba-tiba, jam operasional tidak update. Satu pengalaman buruk bisa viral negatif.
→ Mitigasi: Local curator per kota yang verifikasi data mingguan, sistem pelaporan user, SMS otomatis ke vendor saat ada perubahan.
Tinggi
Traveloka/Tiket.com clone fitur itinerary
Kompetitor besar bisa meniru fitur inti dalam 3–6 bulan dengan modal yang jauh lebih besar.
→ Mitigasi: Bangun data moat secepat mungkin. Network effect dari vendor & user data yang besar adalah pertahanan terbaik.
Sedang
AI hallucinate destinasi wisata
GPT merekomendasikan tempat yang tidak ada, sudah tutup, atau dengan info yang salah.
→ Mitigasi: Semua output AI divalidasi terhadap database internal sebelum ditampilkan. Gunakan RAG (Retrieval-Augmented Generation) bukan pure generation.
Sedang
Biaya API AI membengkak
Saat scale ke 50K+ MAU, biaya OpenAI API bisa mencapai puluhan juta rupiah per bulan.
→ Mitigasi: Caching itinerary populer, rate limiting per user tier, mulai fine-tune open-source model di fase 3.
Sedang
Regulasi dan perizinan destinasi pemerintah
Taman Nasional, Cagar Budaya, dan destinasi BUMN butuh izin formal yang prosesnya panjang.
→ Mitigasi: Mulai dari destinasi swasta dan komunitas. Masuk destinasi pemerintah via program Kemenparekraf di fase 2.
Rendah
User churn setelah satu perjalanan
User menggunakan Trabas untuk liburan ke Yogyakarta, lalu tidak kembali karena tidak ada perjalanan lagi.
→ Mitigasi: Konten inspirasi perjalanan, push notif pre-long weekend, program loyalty poin, fitur "wishlist destinasi".
Revenue stream per fase
  • Fase 0–1: 0 (investasi)
  • Fase 1: Komisi tiket + Premium sub
  • Fase 2: + B2B deals + API license
  • Fase 3: + SaaS vendor + data insights
  • Fase 4: + regional + insurance komisi
Unit economics target (fase 2)
  • CAC (Cost per User): < Rp 15.000
  • LTV (lifetime value): > Rp 150.000
  • LTV:CAC rasio: > 10:1
  • Gross margin: > 60%
  • Monthly churn: < 8%
Struktur biaya terbesar (yang harus diawasi)
  • Gaji tim teknologi (terbesar — jangan pelit di sini)
  • Biaya API AI (naik seiring scale, harus dimonitor ketat)
  • Customer support & vendor management
  • Marketing & influencer budget
  • Infrastructure cloud (AWS/GCP)
Kebutuhan modal per fase (estimasi)
  • Fase 0 (bootstrapped): Rp 200–500 juta (founder + angel)
  • Fase 1 (pre-seed): Rp 1–3 miliar
  • Fase 2 (seed): Rp 5–15 miliar
  • Fase 3 (Series A): Rp 30–80 miliar
  • Fase 4 (Series B): USD 5–20 juta
⚠ Default to revenue positivity ASAP. Jangan bakar uang untuk growth sebelum unit economics terbukti positif.
